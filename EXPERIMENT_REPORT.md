# 阿里云百炼 vs 本地 Qwen3-VL-Embedding-8B 向量兼容性实验报告

**实验日期**: 2026-07-11  
**实验目的**: 验证阿里云百炼线上 `qwen3-vl-embedding` 服务与本地部署的 `Qwen/Qwen3-VL-Embedding-8B` 模型产生的 embedding 向量是否在同一向量空间中，能否混合使用。

## 2. 实验方法

### 2.1 向量获取方式

**本地模型**:
```python
from qwen3_vl_embedding import Qwen3VLEmbedder
model = Qwen3VLEmbedder(model_name_or_path=MODEL_PATH, torch_dtype=torch.bfloat16)

# 获取原始 4096 维向量（未归一化）
raw = model.process([{"text": text, "instruction": instruction}], normalize=False)
# 截取前 N 维后 L2 重新归一化
vec = l2_normalize(raw[:dim])
```

**百炼 API**:
```python
payload = {
    "model": "qwen3-vl-embedding",
    "input": {"contents": [{"text": text}]},
    "parameters": {"dimension": 2048, "instruct": instruction}
}
resp = httpx.post(DASHSCOPE_URL, json=payload, headers=headers)
vec = resp.json()["output"]["embeddings"][0]["embedding"]
```

### 2.2 一致性度量

- **Cosine Similarity**: 衡量两个向量的方向一致性，范围 [-1, 1]。cos ≈ 1 表示完全一致，cos ≈ 0 表示无相关性。
- **L2 Norm**: 检查向量是否已归一化（应为 1.0）。
- **前 N 维数值直接对比**: 检查向量具体数值是否接近。
- **语义排序一致性**: 对同一组查询和候选项，两端的相似度排名是否一致。

### 2.3 测试场景设计（7 大场景，30+ 项测试）

| 场景 | 说明 | 测试项数 |
|------|------|---------|
| 1. 纯文本 Embedding 对比 | 中文/英文/混合/长短/特殊字符等 | 12 |
| 2. 纯图片 Embedding 对比 | 5 张不同类型测试图片 | 5 |
| 3. 跨模态文图检索排序 | 5 条查询 × 5 张图片的排序一致性 | 5 |
| 4. 不同 Instruction 对比 | 默认/中文/通用/空 instruction | 4 |
| 5. 不同维度对比 | 256 / 512 / 768 / 1024 / 2048 | 5 |
| 6. 语义一致性对比 | 近义/远义/跨语言/改写 | 5 |
| 7. 重复性测试 | 同一输入多次调用 | 2 |

## 4. 关键发现

### 4.1 结论：向量空间不兼容

| 指标 | 数值 | 含义 |
|------|------|------|
| 文本向量 Cosine (12项平均) | **≈ -0.01** | 无相关性 |
| 图片向量 Cosine (5项平均) | **≈ -0.01** | 无相关性 |
| 所有维度的 Cosine 范围 | **-0.056 ~ 0.022** | 均接近 0 |
| 向量前 5 维数值 | 符号/大小完全不同 | 不同空间 |

**本地 `Qwen3-VL-Embedding-8B` 与百炼 `qwen3-vl-embedding` 的向量处于完全不同的向量空间，不能混合使用。**

### 4.2 两端各自工作正常

| 验证项 | 本地 | 百炼 API |
|-------|------|---------|
| 确定性 | cos=1.0 (多次调用) | cos=1.0 (多次调用) |
| 语义近判别 | sim=0.74~0.89 | sim=0.79~0.91 |
| 语义远判别 | sim=0.08~0.13 | sim=0.06~0.22 |
| 跨模态 Top1 | 5/5 正确 | 5/5 正确 |

### 4.3 不兼容的可能原因

1. **模型权重不同**: 百炼线上可能使用了不同规模或额外微调/蒸馏的版本，而非直接部署 HuggingFace 上公开的 8B 权重。
2. **原始向量维度不同**: 本地模型输出 4096 维，百炼最大支持 2560 维 —— 底层隐藏层大小可能不同。
3. **预处理/后处理差异**: API 在内部可能有额外的向量变换（旋转、投影等）。
4. **默认 Instruction 行为不同**: 本地传空 instruction 时回退到固定值，API 传空时使用不同的默认值。

### 4.4 额外发现

- 百炼 API 的不同维度向量确实是从最大维度 (2560) 截取后重新归一化得到的（cos=1.0）。
- 尽管向量空间不同，两个模型的**语义排序方向相似** —— 跨模态 Top1 检索结果全部一致，说明它们训练目标一致，但参数空间不同。

## 5. 实际影响与建议

### 不可以做的事

- 用本地模型对图片建库，再用百炼 API 的文本向量来检索（或反过来）
- 将两端的向量混存在同一个向量数据库中

### 建议方案

| 场景 | 推荐方案 |
|------|---------|
| 全量数据入库 + 在线查询 | **统一使用百炼 API**，一致性有保障 |
| 离线批量处理 | **统一使用本地模型**，节省 API 成本 |
| 需要本地+线上混合 | **不可行**，必须选择其一 |
